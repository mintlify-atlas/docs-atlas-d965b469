Dify's built-in logs show individual request summaries. Tracing integrations go deeper: they export the full call chain — every LLM call, tool invocation, retrieval step, and moderation check — as structured spans to a third-party observability platform. This gives you flame graphs, latency breakdowns, token attribution per step, and cost analysis across runs.

## What tracing captures

When a tracing integration is active, Dify emits the following span types:

| Span type | What it covers |
|---|---|
| `message` | The complete message exchange: input, output, model, token counts, and latency |
| `workflow` | A full workflow run: all node inputs/outputs, elapsed time, status, and total tokens |
| `tool` | A single tool call: tool name, inputs, outputs, time cost, and any error |
| `dataset_retrieval` | A knowledge base lookup: query, retrieved documents, and timing |
| `moderation` | A moderation check: input, flagged status, action taken, and preset response |
| `suggested_question` | Suggested follow-up generation: model used, tokens, and the generated suggestions |
| `generate_conversation_name` | Automatic conversation title generation |

Spans are batched and dispatched asynchronously via Celery, so tracing adds no latency to your app's critical path.

## Configuring a tracing integration

Tracing is configured per app. Each app can have one active tracing provider at a time.

<Steps>
  <Step title="Open app tracing settings">
    In the Dify console, open your app and go to **Monitoring → Tracing**.
  </Step>

  <Step title="Choose a provider and enter credentials">
    Select your provider and fill in the required credentials (see per-provider details below). Click **Test** to verify the connection before saving.
  </Step>

  <Step title="Enable tracing">
    Toggle **Tracing** to on. Traces will be emitted for all new requests. Existing log entries are not back-filled.
  </Step>
</Steps>

<Note>
  Credentials are encrypted at rest using Dify's tenant-scoped key before being stored. The console displays obfuscated values (masked with `*`) after saving.
</Note>

## What data is sent to tracing providers

Dify sends the following data to tracing providers for each span:

- **Inputs and outputs** — The exact text or structured data that entered and exited each step
- **Start and end timestamps** — ISO 8601 datetimes for span duration calculation
- **Token counts** — Prompt tokens, completion tokens, and total tokens where applicable
- **Model metadata** — Provider name and model ID
- **Status and error messages** — Whether the step succeeded, and the error string if it failed
- **File references** — URLs to any files attached to the message (resolved using the `FILES_URL` environment variable)
- **Conversation and workflow IDs** — For correlation across spans
- **Streaming metrics** — Time to first token and total generation time for streaming requests

<Warning>
  Inputs and outputs sent to tracing providers include the full text of user messages and model responses. Review your tracing provider's data processing agreement before enabling tracing in environments that handle sensitive or regulated data.
</Warning>

## Use cases for tracing in production

- **Latency debugging** — Compare span durations to identify which step in a complex workflow is slowest. A tool call taking 3 seconds is immediately visible in a flame graph.
- **Prompt regression testing** — Capture a representative sample of production traces and replay them against a new prompt version to compare outputs.
- **Cost attribution** — Break down token spend by node or step within a workflow to find optimization opportunities.
- **Error root-cause analysis** — Trace a failed conversation back through every intermediate step to find where it went wrong.
- **Dataset creation** — Export traced inputs and outputs to build evaluation datasets for systematic testing.
