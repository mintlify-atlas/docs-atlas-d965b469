---
title: "Run Workflow"
description: "POST /workflows/run — Execute a workflow app and retrieve its outputs."
---

<Note>
  This endpoint is only available for **Workflow** app types.
</Note>

## Endpoint

```
POST /workflows/run
```

## Request

### Headers

| Header | Value |
|---|---|
| `Authorization` | `Bearer {api_key}` |
| `Content-Type` | `application/json` |

### Body Parameters

<ParamField body="inputs" type="object" required>
  Key-value pairs matching the workflow's defined input variables. Pass `{}` if the workflow has no inputs.
</ParamField>

<ParamField body="user" type="string" required>
  A unique end-user identifier for analytics and attribution.
</ParamField>

<ParamField body="response_mode" type="string">
  How the response is delivered.

  - `"blocking"` (default) — waits for the full workflow to complete and returns all outputs.
  - `"streaming"` — returns a Server-Sent Events stream with per-node progress events.
</ParamField>

<ParamField body="files" type="array">
  Files to pass into the workflow as file-type input variables.

  <Expandable title="File object">
    <ParamField body="type" type="string" required>
      File type. One of: `"image"`, `"document"`, `"audio"`, `"video"`, `"custom"`.
    </ParamField>
    <ParamField body="transfer_method" type="string" required>
      `"remote_url"` or `"local_file"`.
    </ParamField>
    <ParamField body="url" type="string">
      Publicly accessible URL. Required when `transfer_method` is `"remote_url"`.
    </ParamField>
    <ParamField body="upload_file_id" type="string">
      ID from the [Upload File](/api-reference/files/upload-file) endpoint. Required when `transfer_method` is `"local_file"`.
    </ParamField>
  </Expandable>
</ParamField>

## Response

### Blocking mode

<ResponseField name="workflow_run_id" type="string">
  Unique identifier for this workflow run. Use it with [Get Run](/api-reference/workflow/get-run) to retrieve results later.
</ResponseField>

<ResponseField name="task_id" type="string">
  Task identifier for stopping a streaming run. Not meaningful for blocking mode.
</ResponseField>

<ResponseField name="data" type="object">
  Workflow execution result.

  <Expandable title="Data fields">
    <ResponseField name="id" type="string">Workflow run ID.</ResponseField>
    <ResponseField name="workflow_id" type="string">ID of the workflow definition that was executed.</ResponseField>
    <ResponseField name="status" type="string">
      Execution status. One of: `"succeeded"`, `"failed"`, `"stopped"`, `"running"`, `"paused"`.
    </ResponseField>
    <ResponseField name="inputs" type="object">The inputs that were provided to the workflow.</ResponseField>
    <ResponseField name="outputs" type="object">The outputs produced by the workflow.</ResponseField>
    <ResponseField name="error" type="string">Error message if status is `"failed"`.</ResponseField>
    <ResponseField name="total_steps" type="integer">Total number of workflow nodes executed.</ResponseField>
    <ResponseField name="total_tokens" type="integer">Total LLM tokens consumed during the run.</ResponseField>
    <ResponseField name="elapsed_time" type="number">Total execution time in seconds.</ResponseField>
    <ResponseField name="created_at" type="integer">Unix timestamp of when the run started.</ResponseField>
    <ResponseField name="finished_at" type="integer">Unix timestamp of when the run completed. `null` if still running.</ResponseField>
  </Expandable>
</ResponseField>

### Streaming mode

Returns SSE events. Key event types:

| Event | Description |
|---|---|
| `workflow_started` | Workflow execution has begun; includes `workflow_run_id` and `task_id` |
| `node_started` | A workflow node has started; includes `node_id`, `node_type`, `title` |
| `node_finished` | A workflow node has finished; includes `outputs`, `status`, `elapsed_time` |
| `workflow_finished` | Workflow complete; includes final `data` matching the blocking response shape |
| `error` | A fatal error occurred |
| `ping` | Keep-alive heartbeat |

## Error Codes

| HTTP Status | Code | Description |
|---|---|---|
| `400` | `invalid_param` | Missing or invalid input parameters |
| `401` | `unauthorized` | Invalid or missing API key |
| `404` | `not_found` | Workflow definition not found |
| `429` | `rate_limit` | Rate limit exceeded |
| `500` | `provider_not_initialize` | LLM provider credentials not configured |
| `500` | `provider_quota_exceeded` | LLM provider quota exhausted |
| `500` | `model_currently_not_support` | Selected model unavailable |
| `500` | `completion_request_error` | LLM provider returned an error |

## Examples

<CodeGroup>
```bash curl (blocking)
curl -X POST 'https://api.dify.ai/v1/workflows/run' \
  -H 'Authorization: Bearer {api_key}' \
  -H 'Content-Type: application/json' \
  -d '{
    "inputs": {"article": "Dify is an open-source LLM app development platform."},
    "response_mode": "blocking",
    "user": "user-123"
  }'
```

```python Python (blocking)
import requests

response = requests.post(
    "https://api.dify.ai/v1/workflows/run",
    headers={"Authorization": "Bearer {api_key}"},
    json={
        "inputs": {"article": "Dify is an open-source LLM app development platform."},
        "response_mode": "blocking",
        "user": "user-123",
    },
)
print(response.json())
```

```bash curl (streaming)
curl -X POST 'https://api.dify.ai/v1/workflows/run' \
  -H 'Authorization: Bearer {api_key}' \
  -H 'Content-Type: application/json' \
  -H 'Accept: text/event-stream' \
  --no-buffer \
  -d '{
    "inputs": {"article": "Dify is an open-source platform."},
    "response_mode": "streaming",
    "user": "user-123"
  }'
```
</CodeGroup>
